# _local-edr-research-001: AI Application Testing Taxonomy

## Abstract

AI applications add non-deterministic output, autonomous tool use, and multi-node orchestration on top of conventional software. agentme's existing engineering decisions (agentme-edr-007, 018, 019, 020, 021, 028) already prescribe testing mechanics for parts of this stack, but no single document catalogs the available test types, when each runs, or their relevance to engineering and business. This research synthesizes agentme's testing-related EDRs with three external references — Hamel Husain's three-level LLM evaluation model, the OWASP Top 10 for LLM Applications, and Microsoft Foundry's generative-AI observability lifecycle — into a curated taxonomy of 12 test types across 5 groups: code-level, prompt/LLM, quality eval, safety/adversarial, and responsible AI (fairness, bias, robustness, explainability). Each entry records its test objective, techniques, applicable tier, execution level (unit, integration, or production), whether mocks are usable, when the test is required, which other test lowers that need, and priority for the development team and for the business. The synthesis shows agentme's current EDRs cover mocked unit testing and dataset-driven quality evals, but not safety, adversarial, or responsible-AI tests. The taxonomy is evidence for a future EDR that closes these gaps.

## Introduction

AI-based components are built at different levels of abstraction, and agentme already defines three tiers for this: **LLM** (a single request/response call to a model), **Agent** (an LLM-driven tool-invocation loop), and **Workflow** (a directed graph mixing LLM, agent, and algorithmic nodes) — see agentme-edr-018. Each tier introduces testing concerns classic unit and integration testing were not designed for: non-deterministic outputs, prompts that drift in behavior across model versions, autonomous tool selection, and multi-step state machines that can fail at any node.

agentme already carries testing guidance for parts of this stack, but it is spread across several documents rather than named as a single catalog:

- agentme-edr-004 and agentme-edr-007 (rules 02, 04, 08) set the baseline for conventional unit and integration tests.
- agentme-edr-007 rule 09 gives a per-tier requirement table (unit tests / evals / integration tests, by LLM / Agent / Workflow tier) but names only these three coarse categories.
- agentme-edr-018 and agentme-edr-021 define concrete mocking patterns for offline unit tests (`FakeListChatModel`, `GenericFakeChatModel`) and prompt/output-length conventions.
- agentme-edr-028 defines the folder layout, script requirements, and report format for dataset-driven evals.

None of these documents name or position test types such as LLM-as-judge scoring, human evaluation, RAG-specific groundedness metrics, adversarial or red-team testing, safety/content evals, or fairness/bias/robustness/explainability testing — all established practice in the wider industry per the external sources reviewed in this study (see Methods). A first pass of this research also surfaced agent-trajectory, task-completion, and release/production test types (A/B testing, drift monitoring); the requester then curated the final table down to the tests judged most decision-relevant for day-to-day development and business review, and asked for a dedicated Responsible-AI group to be added, rather than keeping the full initial list.

**Constraints and assumptions:**

- This study treats "API" (the term requested for the output table) as synonymous with agentme's existing "LLM" tier — a raw request/response call to a model — to stay consistent with the vocabulary already used across agentme's AI EDRs.
- "Prompts" is treated as a cross-cutting technique concern (prompt content, versioning, and prompt-specific assertions) that can occur inside any of the LLM, Agent, or Workflow tiers, rather than as a fourth peer architectural tier.
- Not every test in the table is meant to run for every Prompt, Agent, or Workflow: rows that differ only in tier (a plain-code vs. a workflow-node unit test) or in scoring technique (an LLM-as-judge rubric vs. a similarity-vector or exact-match score) were merged into a single row with the choice documented in Techniques; the When it's Required column states the condition that triggers each remaining row instead of relying on a cross-row alternatives flag.
- The study is a qualitative synthesis of a fixed, disclosed evidence set (agentme's own EDRs plus three external references); it does not attempt an exhaustive survey of the eval-tooling market.
- Per direction from the requester, the immediate goal is to produce evidence that can later support a new EDR proposing a fuller AI-testing taxonomy for agentme; this document does not itself change any existing Policy.

The objective of this research is to produce one reusable table naming each relevant test type, grouping related types together, and stating each type's test objective, techniques, applicable tier, execution level, mocking constraints, when it is required, and priority for the development team and for the business.

Question: Which test types should agentme's AI testing standards name explicitly, where does each one apply, which are interchangeable alternatives rather than additive requirements, and why does each matter to engineers and to the business?

## Methods

The taxonomy was built in five reproducible steps.

**Step 1 — Enumerate internal evidence.** Read every agentme EDR whose scope covers testing or AI development: agentme-edr-004 (unit test requirements), agentme-edr-007 (project quality standards, especially rule `09-ai-project-testing-requirements`), agentme-edr-018 (LLM development standards: tier model, mocking, prompt management), agentme-edr-019 (agent structural patterns), agentme-edr-020 (agent quality standards: tool definitions, error handling, unit testing), agentme-edr-021 (workflow development standards: verification nodes, unit testing, evals), agentme-edr-028 (eval standards: folder layout, script and report requirements), and agentme-edr-029 (naming conventions, including the `evaluate_*` judge-node prefix). Every rule that named or implied a distinct test or eval type was extracted into a working list together with its source rule number.

**Step 2 — Gather external evidence.** Three external references were selected for being widely cited and for covering complementary angles not found internally:

1. Hamel Husain, *"Your AI Product Needs Evals"* (hamel.dev, March 2024) — a practitioner-tested three-level model (Level 1 unit-test assertions, Level 2 human and model/LLM-as-judge evaluation, Level 3 A/B testing), plus a note on evaluating RAG separately.
2. OWASP Top 10 for LLM Applications / GenAI Security Project (owasp.org, genai.owasp.org) — the adversarial and security-testing angle (prompt injection, jailbreaks, unsafe tool use).
3. Microsoft Foundry generative-AI observability documentation (learn.microsoft.com) — a three-stage AI lifecycle (base model selection, pre-production evaluation, post-production monitoring) and its built-in evaluator categories (general quality such as coherence/fluency, RAG-specific such as groundedness/relevance, safety/security such as hate/violence/protected material, and agent-specific such as tool-call accuracy/task completion), including its PyRIT-based automated red-teaming agent.

**Step 3 — Normalize vocabulary.** Every test type found in step 1 or step 2 was mapped onto the "Where to Apply" categories requested for the output table: API, Prompts, Agents, Workflows. API was aligned to agentme's existing "LLM" tier (agentme-edr-018) so the table stays consistent with agentme's own vocabulary; Prompts was treated as a cross-cutting technique dimension that can co-occur with any of the other three, per the assumption stated in the Introduction.

**Step 4 — Deduplicate and group.** Near-duplicate entries were merged only when they described the same test at the same level of abstraction. An initial pass produced 19 entries across 7 groups, from most foundational (code-level) to most operational (release and production). The requester then curated this list: dropped the three Agent-specific rows and the two Release & production rows as out of scope, kept the remaining ten in a business/engineering-relevance-first order, and requested four Responsible-AI test types (fairness, bias, robustness, explainability) as a new group. The final table had 14 rows across 6 groups, ordered from the highest-stakes concerns down to foundational code-level tests.

**Step 5 — Populate the table.** For every entry, seven fields were recorded: Objective (what the test verifies), Techniques (concrete tools/patterns, preferring ones already named in agentme's EDRs, e.g. `FakeListChatModel`, `GenericFakeChatModel`, Wilson score CI), Where to Apply (one or more of the four categories), Selection Group (alternatives vs. independent/additive entries), Relevance for Development Team (e.g. fast feedback, catching a regression on a model/provider swap), and Relevance for Business (e.g. a regulatory requirement or business-correctness guarantee). A second, smaller table cross-checked each group against agentme's existing EDRs (Yes/Partially/No, with citation) so the coverage gap is inspectable in Results.

**Step 6 — Consolidate rows that differ only by tier or scoring technique.** After reviewing the 14-row table, the requester asked for two further merges: the Code-level unit test and the Workflow-specific mocked node unit test were folded into one **Unit test (offline, mocked)** row, since both use the same offline/mocked technique and differed only in tier; and the LLM-as-judge eval and Golden-dataset accuracy eval were folded into one **Golden-dataset accuracy / LLM-as-judge eval** row, with the choice documented in Techniques instead of as two rows. The final table has 12 rows across 5 groups; Workflow no longer appears as a distinct group.

**Step 7 — Classify execution stage and mocking constraints.** Each row was further classified by Level — Unit Tests (offline, mocked, scoped to one component/agent/workflow, no external dependencies), Integration Tests (real LLM/provider and external-service calls, no mocking), or Production Tests (runs against live traffic as an A/B test or shadow run) — allowing more than one value where the same test type is legitimately run at more than one stage (e.g. a quality eval used both pre-release and as continuous production monitoring). A parallel Can Use Mocks field records Yes (fully mocked, e.g. the Unit test row), No (no mocking of any kind, e.g. the Integration test row), or Not for LLM calls (surrounding harness code may be simplified, but the model/agent/workflow call itself must be real for the test to be meaningful).

**Step 8 — Rename, remove, and add columns per requester feedback.** Objective was renamed **Test Objective**. Selection Group was removed — after Step 6's merges every row had become Independent, so it no longer distinguished any row — and replaced with **When it's Required**, grounded in agentme-edr-007 rule 09 for the two rows it already governs (required for Workflow) and in professional judgment for the rest. **Priority for Team** and **Priority for Business** were added with Low/Medium/High ratings, giving the Relevance columns a comparable severity label.

**Step 9 — Map redundancy between rows.** Each row was checked against every other for genuine overlap (e.g. a comprehensive Fairness test already surfaces what a Bias test targets); the reducing row's name was recorded in a new **Need lowered if this other test type is performed** column, or None when no overlap exists.

**Step 10 — Rate implementation difficulty.** Each row was rated Low, Medium, or High in a new **Difficulty to Implement** column, based on tooling maturity: Low where agentme already has a concrete pattern or the technique is classic software engineering, Medium where external tooling exists but needs integration and calibration, and High where the metric itself is methodologically hard or still evolving (e.g. feature-attribution faithfulness, disparate-impact analysis).

**Limitations.** No statistical or experimental validation was performed; this is a qualitative Policy-and-literature synthesis, consistent with the evidence bar research documents are expected to meet. The external evidence set is three references, not an exhaustive survey; frameworks like promptfoo, DeepEval, RAGAS, TruLens, or Arize Phoenix were not reviewed. Grouping boundaries (step 4) and the "Prompts as cross-cutting" normalization (step 3) are judgment calls, stated here so a future author can revisit them. The curation that dropped Agent-specific and Release & production rows narrows this pass to the tests judged most relevant to day-to-day development and business review; those concerns remain valid, just out of scope for this iteration.

## Results

### Test type taxonomy

Following curation by the requester, 12 test types are organized into 5 groups, ordered from the highest-stakes concerns (safety, adversarial, and responsible AI) down to foundational code-level tests. Two pairs of rows that differed only by tier or by scoring technique were merged: a plain-code unit test and a workflow-node unit test are now one **Unit test (offline, mocked)** row, and an LLM-as-judge eval and a golden-dataset accuracy eval are now one **Golden-dataset accuracy / LLM-as-judge eval** row, with the technique choice documented in Techniques. "Where to Apply" uses the requested categories — API, Prompts, Agents, Workflows — and lists more than one value when a test type recurs across tiers. "Level" classifies each row as Unit Tests, Integration Tests, or Production Tests per the definitions in Methods, and lists more than one value when a test type runs at more than one stage; "Can Use Mocks" records whether mocking is acceptable for that row (Yes, No, or Not for LLM calls). "When it's Required" states the condition that triggers each row, grounded in agentme-edr-007 rule 09 where that policy already applies and in professional judgment elsewhere. "Priority for Team" and "Priority for Business" rate each row Low, Medium, or High from the development team's and the organization's point of view respectively. "Need lowered if this other test type is performed" names another row whose thorough execution reduces this row's marginal value, or "None" when no such overlap exists. "Difficulty to Implement" rates the same Low/Medium/High scale for the effort to build and operate the row.

The remaining ten rows are advised or recommended. Priority is High for both team and business on those two rows plus Safety/content eval and RAG groundedness/faithfulness eval; Explainability test is the only row rated Low for team priority even though it rates High for Difficulty to Implement. Golden-dataset accuracy / LLM-as-judge eval also lowers the marginal need for four other rows, while Bias test's need is lowered by Fairness test and Robustness test's by Adversarial / red-team test.

### Mocked unit test vs. dataset eval — a recurring trade-off

A recurring decision point is whether to verify a component with a mocked, offline unit test or a real-provider dataset eval — now represented by exactly the two general-purpose rows at the foundation of the table, Unit test (offline, mocked) and Golden-dataset accuracy / LLM-as-judge eval. The trade-off generalizes to any tier:

| Option | Pros | Cons |
|---|---|---|
| Unit test (offline, mocked) | Free, deterministic, runs on every commit (agentme-edr-004 rule 02) | Cannot detect prompt/model quality regressions or drift |
| Golden-dataset accuracy / LLM-as-judge eval (real provider) | Catches real quality issues and model drift; produces auditable metrics and thresholds (agentme-edr-028) | Costs time/money; pass rate is not deterministic; needs a maintained dataset |

agentme-edr-007 rule 09 already requires both, in combination, for the Workflow tier; the same combination is good practice for any Prompt or Agent critical enough to warrant it, even though rule 09 does not currently require it there. The Level and Can Use Mocks columns make this pattern explicit for every row: Unit test (offline, mocked) is Level = Unit Tests / Can Use Mocks = Yes, while Golden-dataset accuracy / LLM-as-judge eval is Level = Integration Tests, Production Tests / Can Use Mocks = Not for LLM calls. Both are also the only two rows marked Required in When it's Required, since they are the two tests agentme-edr-007 rule 09 already mandates for the Workflow tier.

## Discussion

The taxonomy confirms agentme's current AI EDRs concentrate on the earliest, cheapest layers of the testing pyramid: mocked unit tests on every commit, whether over plain code or workflow-node routing (agentme-edr-004, agentme-edr-018 rule 04, agentme-edr-021 rule 10), and dataset-driven quality evals with explicit thresholds (agentme-edr-028). That emphasis matches the "iterate fast" principle central to both Hamel Husain's model (offline assertions first) and agentme-edr-004's preference for offline, deterministic tests. It also means the two groups with no internal coverage — Safety & adversarial and Responsible AI — are not oversights so much as a natural frontier: exactly the layers classic software engineering has less established practice for, and that the external references were chosen to cover.

Safety and adversarial testing remains the most consequential gap: agentme's agent and workflow EDRs (agentme-edr-019, agentme-edr-020) already describe the exact surface that OWASP's LLM Top 10 and Microsoft's PyRIT-based red-teaming agent target — autonomous tool-invocation loops manipulable through prompt injection or unsafe tool calls. Shipping detailed agent/workflow guidance without any adversarial-test guidance leaves a known risk surface unaddressed.

Adding Responsible AI as its own group, rather than folding fairness/bias/robustness/explainability into Safety & adversarial, is a defensible split: adversarial and content-safety tests probe for a hostile actor or unsafe output, while fairness, bias, robustness, and explainability tests probe ordinary behavior for equity, stability, and accountability even absent any attack. Bias and fairness overlap in practice — a stereotyped-association metric is often one input to a fairness judgment such as disparate impact — so a team adopting this table should expect to reuse a shared harness across the two rather than build them independently.

The consolidation itself is a finding: two candidate rows differed only by tier or scoring technique, not by underlying purpose, so keeping them separate would have overstated how many distinct testing concerns exist. Merging them left every remaining row without a genuine alternative, which is why Selection Group was replaced with When it's Required and the two Priority columns — dimensions that add new information for every row.

The Relevance columns make the taxonomy directly usable for prioritization conversations mixing engineering and business stakeholders. Several rows justify themselves almost entirely on regulatory or reputational grounds (Fairness test, Bias test, Safety/content eval) rather than defect-catching value \u2014 a different argument than the one that normally justifies unit and integration tests; a future EDR should keep both kinds of justification visible rather than collapsing all AI tests into a single "quality" rationale.

The Level column also partially answers the requester's earlier decision to drop Release & production test types (A/B testing, drift monitoring) as separate rows: nine of the twelve rows can also run at the Production Tests level, since the same eval logic commonly runs both pre-release and continuously against live traffic (per Microsoft Foundry's pre-production/post-production split). This does not reintroduce A/B testing as a distinct row — comparing two live variants is a genuinely different mechanism — but shows most evals here are stage-agnostic rather than pre-release-only.

The Priority columns surface a recurring tension: the three Responsible AI rows rate only Medium or Low for the team but High or Medium for the business, since their justification is regulatory and reputational rather than defect-catching. The reverse occurs for Prompt regression test: High for the team (cheap, catches everyday edits) but only Medium for the business. Teams prioritizing by engineering convenience alone risk under-investing in Responsible AI; teams prioritizing by business risk alone risk under-investing in the fast, cheap checks that catch most day-to-day regressions.

The evidence base has real limits. Three external references cover complementary angles but are not an exhaustive market survey; frameworks like promptfoo, DeepEval, RAGAS, and TruLens were not reviewed, and fairness/robustness/explainability metrics were not benchmarked. Difficulty ratings are qualitative, not derived from measured engineering hours. The requester's curation also removed Agent-specific and Release & production test types from the primary table; their tiers remain represented through Where to Apply, but a future EDR should treat this table as a starting subset, not a complete taxonomy.

## Conclusion

This study catalogs 12 AI-application test types across 5 groups — code-level, prompt/LLM, quality eval, safety and adversarial, and a responsible-AI group covering fairness, bias, robustness, and explainability — and marks where each applies among the API (LLM), Prompts, Agents, and Workflows categories used across agentme. A Level column classifies each row as Unit, Integration, or Production Tests, a Can Use Mocks column records whether mocking is acceptable, a When it's Required column states the condition that triggers each test, and Priority for Team / Priority for Business columns rate each row Low, Medium, or High for engineering and organizational stakeholders respectively.

Cross-checking the table against agentme's existing engineering decisions shows three groups with full or partial internal coverage (Code-level, Prompt/LLM, Quality eval) and two with none: Safety & adversarial and Responsible AI. A Need lowered if this other test type is performed column records cross-row redundancy — e.g. Bias test's need is lowered by a thorough Fairness test, and four rows are lowered by a thorough Golden-dataset accuracy / LLM-as-judge eval.

The practical takeaway is that the highest-leverage next addition to agentme's AI testing standards is safety/content and adversarial/red-team guidance, followed by a Responsible AI rule set (fairness, bias, robustness, explainability), because both gaps carry regulatory and reputational relevance that is independent of defect-catching value and is not covered by any existing EDR.

This taxonomy is a qualitative synthesis intended as evidence for a future EDR, not a certified benchmark or a final rule set; it reflects one curated, business/engineering-relevance-first ordering of a broader initial list, and the dropped Agent-specific and Release & production test types remain valid topics for a separate or later pass.

Open questions for the follow-up EDR: (1) which concrete metrics and tooling agentme should standardize on for fairness, bias, robustness, and explainability testing; (2) which concrete adversarial-testing approach agentme should standardize on (a maintained attack-pattern dataset vs. an automated red-teaming agent such as PyRIT); (3) whether Agent-specific tests (trajectory, tool-call accuracy, task completion) and Release & production tests (A/B testing, drift monitoring) should be folded back into the same EDR or kept as separate follow-ups; and (4) how the When it's Required, Priority, and Relevance columns should be represented in a future EDR's rule structure without duplicating the table itself.
